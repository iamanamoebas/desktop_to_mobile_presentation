Architecture

How will we code the site?

  • Separate Code Base
  • Responsive
* separate domains - `m.` * Dynamic switch - same domain, separate code bases * Responsive - 1 code base for all the things

Separate Code Base

  • Keeps each code base simpler
  • Smaller File sizes individually
  • Faster to code if new to mobile
  • Easier to make decisions
  • Less to break when updating
  • Easier to tailor experience
  • Allows for developers to "Specialize" in one base
* tailor experience - like, "this report is unnecessary on mobile", remove it. * decisions - like, "add a print button"?

Responsive Code Base

  • Requires good code organization
  • Smaller file sizes combined
  • More likely to net a better product in the end
  • One code base to maintain
  • Faster to code if familiar with UI
  • Generally accepted as the "recommended" method
* good for sites that will likely be the same on both. * experienced devs only

Images

Where possible, use vectors!

SVG is Fully Suported

SVG > Icon Font (usually)

  • More Accessible
  • More Flexible
  • Multi-color
- More accessible (some users have forced fonts, so icons don't look weird if svg) - more flexible (don't have to be in before or after, can be a background) - BUT icon fonts are *slightly* better performance. & can change color easily.

1x, 2x, & 3x Images


            <img src="gob.jpeg"
                 srcset="gob.jpeg 1x,
                         gob@2x.jpeg 2x,
                         gob@3x.jpeg 3x">
1x 2x 3x better supported than specific sizes.

Lazy Load


            <img data-src="gob.jpeg"
                 data-srcset="gob.jpeg 1x,
                         gob@2x.jpeg 2x,
                         gob@3x.jpeg 3x">

Allows for slick animations too!

they will all work about the same, img path goes in a data attr, and detects when to lad the images.
Bonus: can get slick animations if done well.

Events & Animations

Transform

If possible, use transform.

Translate doesn't repaint. hardware accelerated.

Easing

Usually use ease-out

google recommends ease-out because it feels most natural to a user 

// no!
.c-menu {
  position: absolute;
  left: 0;
  transition: left .2s ease;
}
.c-menu.is-open{
  left: 50px;
}
          

// yes!
.c-menu {
  position: absolute;
  left: 0;
  transition: transform .2s ease-out;
}
.c-menu.is-open{
  transform: translateX(50px);
}

click vs tap Events

click is slow, but reliable.

tap (touch) is fast, but not fully supported.

up to 300ms slower.
mouse & touch events also nto the same.

Use a polyfill!

Be Careful...


$('.js-btn')
  .on('pointerup', function(e){
    e.preventDefault(); // doesn't do anything.
  })
  .on('click', function(e){
    e.preventDefault(); // still suppress events this way
  })
